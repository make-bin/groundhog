# Datastore Layer Standards - go-sdk

## Purpose

Use go-sdk's datastore package for all database operations. The go-sdk (`git.eniot.io/apaas/go-sdk`) provides unified database abstraction, transaction management, and CRUD operations.

## Location

```
pkg/infrastructure/datastore/
├── datastore.go          # go-sdk DataStore initialization
└── config.go             # Database configuration
```

## Core Principles

### 1. go-sdk Integration
- Use go-sdk datastore package for ALL database operations
- NO custom database abstraction layer
- NO direct GORM usage in repositories
- Follow go-sdk patterns and conventions

### 2. Repository Implementation
- Inject go-sdk DataStore via dependency injection
- All operations through go-sdk DataStore interface
- Convert go-sdk errors to domain errors at repository boundary
- Pass context through all operations

### 3. Entity Requirements
- All PO must implement go-sdk Entity interface
- Required methods: `TableName()`, `PrimaryKey()`, `Index()`, `SetCreateTime()`, `SetUpdateTime()`, `ShortTableName()`
- Return non-empty values from required methods

## go-sdk DataStore Operations

### CRUD Operations
- `Add(ctx, entity)` - Insert
- `BatchAdd(ctx, entities)` - Batch insert
- `Put(ctx, entity)` - Update
- `Delete(ctx, entity)` - Delete
- `Get(ctx, entity)` - Retrieve by primary key
- `List(ctx, query, options)` - Query with filters/pagination
- `Count(ctx, entity, options)` - Count
- `IsExist(ctx, entity)` - Check existence

### Transaction Operations
- `BeginTransaction(ctx)` - Start transaction, returns context
- `CommitTransaction(ctx)` - Commit
- `RollbackTransaction(ctx)` - Rollback
- `IsTransactionActive(ctx)` - Check active transaction
- `SavePoint(ctx, name)` - Create savepoint
- `RollbackTo(ctx, name)` - Rollback to savepoint
- `ReleaseSavePoint(ctx, name)` - Release savepoint

## Entity Interface (go-sdk)

### Required Methods

All PO must implement:
- `TableName() string` - Table name (non-empty)
- `ShortTableName() string` - Short name for logging
- `PrimaryKey() string` - Primary key value (non-empty)
- `Index() map[string]interface{}` - Index fields for queries
- `SetCreateTime(time.Time)` - Set creation timestamp
- `SetUpdateTime(time.Time)` - Set update timestamp

## Transaction Management

### Rules

**MUST FOLLOW**:
- ✅ Transaction management in application layer ONLY
- ✅ Use context returned by `BeginTransaction()` for all operations
- ✅ Always defer rollback for error handling
- ✅ Commit explicitly on success
- ✅ Pass transaction context through repository calls
- ❌ NO transaction management in repository layer
- ❌ NO manual transaction handling in repositories
- ❌ NO nested `BeginTransaction()` without checking `IsTransactionActive()`

### Pattern

```go
import "git.eniot.io/apaas/go-sdk/pkg/datastore"

// Application layer
txCtx, err := datastore.BeginTransaction(ctx)
defer func() {
    if err != nil {
        datastore.RollbackTransaction(txCtx)
    }
}()

// Operations use transaction context
err = repo.Create(txCtx, entity)
if err != nil { return err }

return datastore.CommitTransaction(txCtx)
```

## Repository Implementation

### Structure

```go
import "git.eniot.io/apaas/go-sdk/pkg/datastore"

type repositoryImpl struct {
    DataStore datastore.DataStore `inject:"datastore"`
}
```

### Operation Mapping

- `Create` → `datastore.Add(ctx, po)`
- `FindByID` → `datastore.Get(ctx, po)` with primary key set
- `Update` → `datastore.Put(ctx, po)`
- `Delete` → `datastore.Delete(ctx, po)` with primary key set
- `List` → `datastore.List(ctx, query, options)` + `datastore.Count()`
- `Exists` → `datastore.IsExist(ctx, po)`

### Rules

**MUST FOLLOW**:
- ✅ Convert domain models to PO before go-sdk operations
- ✅ Convert PO to domain models after go-sdk operations
- ✅ Convert go-sdk errors to domain errors
- ✅ Pass context through all operations
- ✅ Use transaction context from application layer
- ❌ NO direct GORM usage
- ❌ NO transaction management in repositories
- ❌ NO business logic in repositories
- ❌ NO exposing go-sdk errors to domain layer

## Error Handling

### go-sdk Error Types

- `datastore.ErrRecordNotExist` - Record not found
- `datastore.ErrRecordExist` - Duplicate record
- `datastore.ErrPrimaryEmpty` - Empty primary key
- `datastore.ErrTableNameEmpty` - Empty table name
- `datastore.ErrNilEntity` - Nil entity

### Conversion Rules

**MUST convert at repository boundary**:
- `datastore.ErrRecordNotExist` → `domain.ErrEntityNotFound`
- `datastore.ErrRecordExist` → `domain.ErrEntityAlreadyExists`
- Other errors → Log and return `domain.ErrInternalError`

## Query Options (go-sdk)

### ListOptions

- `Page` - Page number (1-based)
- `PageSize` - Items per page
- `SortBy` - Array of sort options (field + order)
- `FuzzyQuery` - Fuzzy search filters
- `InQuery` - IN clause filters
- `IsNotExistQuery` - NULL/empty checks

## Dependency Injection

### DataStore Registration

```go
import "git.eniot.io/apaas/go-sdk/pkg/datastore"

// Server initialization
ds, err := datastore.NewDataStore(config)
container.ProvideWithName("datastore", ds)
```

### Repository Injection

```go
import "git.eniot.io/apaas/go-sdk/pkg/datastore"

type repositoryImpl struct {
    DataStore datastore.DataStore `inject:"datastore"`
}
```

## Best Practices

### ✅ DO

- Use go-sdk DataStore for ALL database operations
- Implement Entity interface correctly on all PO
- Convert errors at repository boundary
- Pass context through all operations
- Use transaction context from application layer
- Leverage go-sdk built-in features

### ❌ DON'T

- Use GORM directly in repositories
- Create custom database connections
- Manage transactions in repositories
- Expose go-sdk errors to domain layer
- Skip Entity interface methods
- Return empty values from `PrimaryKey()` or `TableName()`
- Bypass go-sdk abstraction

## Summary

**Key Rules**:
- ALL database operations through go-sdk DataStore
- NO direct GORM usage in repositories
- Transaction management in application layer ONLY
- PO must implement go-sdk Entity interface
- Convert go-sdk errors to domain errors at repository boundary
- Inject DataStore via dependency injection container
