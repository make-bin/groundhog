# Infrastructure Layer - Migration Standards

## Purpose

Database migrations are infrastructure concerns managed using `golang-migrate/migrate`. This document defines standards for migration management, execution, and integration.

## File Structure

```
pkg/infrastructure/migration/
├── manager.go    # MigrationManager interface and implementation
├── config.go     # MigrationConfig validation and building
├── runner.go     # RunMigrations startup function
├── logger.go     # Logger adapter for golang-migrate
└── source.go     # Migration source abstractions (filesystem, embedded)

migrations/
└── {version}_{description}.{up|down}.sql
```

## Core Principles

### Dependencies
- ✅ `pkg/utils/` (logger, config)
- ✅ `golang-migrate/migrate` library
- ❌ NO domain, application, or interface layers

### Key Rules
- NO business logic in migrations
- Configuration-driven execution
- Fail fast on errors with recovery instructions
- Sanitize passwords in all logs and errors
- Run migrations on application startup

## MigrationManager Interface

**Location**: `pkg/infrastructure/migration/manager.go`

### Methods
- `Up(ctx context.Context) error` - Apply all pending migrations
- `Down(ctx context.Context) error` - Rollback last migration
- `DownTo(ctx context.Context, version uint) error` - Rollback to specific version
- `Version() (version uint, dirty bool, err error)` - Get current state
- `Close() error` - Release resources

### Implementation Requirements
- Context-aware operations for cancellation
- Detect and report dirty state before execution
- Wrap errors with context and recovery instructions
- Sanitize passwords in all error messages
- Structured logging with version and state fields

### Standard Errors
- `ErrMigrationDisabled` - Migrations disabled in config
- `ErrDirtyDatabase` - Failed migration requires manual intervention
- `ErrInvalidSource` - Migration source inaccessible
- `ErrConnectionFailed` - Database connection failed
- `ErrNoChange` - Already up-to-date (not a failure)

## MigrationConfig

**Location**: `pkg/infrastructure/migration/config.go`

### Fields
- `Enabled` (bool) - Enable/disable migrations on startup
- `SourceType` (string) - "filesystem" or "embedded"
- `SourcePath` (string) - Path to migration files
- `DatabaseURL` (string) - Full postgres connection URL
- `LockTimeout` (time.Duration) - Migration lock timeout
- `TableName` (string) - Schema migrations table name

### Validation
- Skip validation if `Enabled` is false
- Validate source type is "filesystem" or "embedded"
- Validate source path and database URL are not empty
- Validate lock timeout is positive
- Return detailed errors for invalid configuration

### Database URL Format
`postgres://user:password@host:port/dbname?sslmode=mode`

Build from database config components (host, port, user, password, dbname, sslmode)

## RunMigrations Function

**Location**: `pkg/infrastructure/migration/runner.go`

**Purpose**: Execute migrations on application startup

### Execution Flow
1. Validate config and logger
2. Build MigrationConfig from app config
3. Exit early if migrations disabled
4. Create migration manager with retry (max 3 attempts, exponential backoff)
5. Check for dirty state (fail if dirty with recovery instructions)
6. Execute `Up()` to apply pending migrations
7. Handle `ErrNoChange` as success (already up-to-date)
8. Log final version and close manager

### Retry Logic
- Max 3 attempts with 2s initial delay, exponential backoff
- Retry only on connection errors: "connection refused", "dial tcp", "no such host", etc.
- Fail immediately on dirty state or validation errors

### Dirty State Recovery
When dirty state detected, provide recovery instructions:
1. Force version if migration partially succeeded
2. Rollback to previous version
3. Check logs and fix manually

### Password Sanitization
Replace passwords with `****` in all error messages and logs using regex pattern

## Migration Sources

**Location**: `pkg/infrastructure/migration/source.go`

### MigrationSource Interface
- `GetSourceURL() string` - Return URL for golang-migrate
- `Validate() error` - Validate source accessibility

### Filesystem Source
- Read from directory on disk
- Validate path exists and is directory
- Convert to absolute path
- Return `file://` URL

### Embedded Source
- Read from `embed.FS` (compiled into binary)
- Register with iofs driver
- Return `iofs://` URL
- Validate filesystem is not nil and contains files

**Usage**:
```go
//go:embed migrations/*.sql
var migrationsFS embed.FS
source := NewEmbeddedSource(migrationsFS, "migrations")
```

## Migration Logger

**Location**: `pkg/infrastructure/migration/logger.go`

Adapter to bridge project logger with golang-migrate's logger interface.

### Implementation
- `Printf(format string, v ...interface{})` - Log at INFO level
- `Verbose() bool` - Return true for verbose logging
- Use structured logging with consistent format

## Migration Files

**Location**: `migrations/`

### Naming Convention
`{version}_{description}.{direction}.sql`

- Version: Sequential (000001, 000002, ...)
- Description: snake_case
- Direction: `up` or `down`

**Examples**:
- `000001_create_users_table.up.sql`
- `000001_create_users_table.down.sql`

### Writing Migrations

**Up Migrations**:
- Create tables, indexes, constraints
- Add columns
- Insert reference data
- Make idempotent when possible
- NO data loss without backup

**Down Migrations**:
- Reverse up migration exactly
- Drop tables, indexes, constraints
- Remove columns
- Must be reversible

**Best Practices**:
- One logical change per migration
- Test both directions
- Use transactions when possible
- Separate schema and data changes
- Document breaking changes

## Application Integration

### Startup Integration
**Location**: `cmd/server/main.go`

Call `migration.RunMigrations(cfg, logger)` after loading config and initializing logger, but BEFORE initializing database connection.

```go
// Run migrations on startup
if err := migration.RunMigrations(cfg, logger); err != nil {
    logger.Fatal("failed to run migrations", "error", err)
}
```

### Configuration
**Location**: `pkg/utils/config/config.go`

Add `MigrationConfig` struct to main config:
```go
type MigrationConfig struct {
    Enabled     bool          `mapstructure:"enabled"`
    SourceType  string        `mapstructure:"source_type"`
    SourcePath  string        `mapstructure:"source_path"`
    LockTimeout time.Duration `mapstructure:"lock_timeout"`
    TableName   string        `mapstructure:"table_name"`
}
```

YAML config example:
```yaml
migration:
  enabled: true
  source_type: filesystem
  source_path: ./migrations
  lock_timeout: 30s
  table_name: schema_migrations
```

- Test password sanitization

### Integration Tests

- Test migration execution (up/down)

## CLI Commands (Optional)

Can create `cmd/migrate/main.go` for manual migration control:
- `migrate up` - Apply all pending
- `migrate down` - Rollback last
- `migrate goto V` - Migrate to version
- `migrate version` - Show current version
- `migrate force V` - Force version (dirty state recovery)

## Logging

Use structured logging with these fields:
- `version` - Current/target version
- `dirty` - Dirty state flag
- `source_type` - Migration source type
- `error` - Sanitized error details

Log levels:
- INFO - Progress, completion
- WARN - Retries, non-critical issues
- ERROR - Failures, dirty state

## Security

### Password Protection
- Sanitize all database URLs in logs and errors
- Replace passwords with `****` using regex
- Never log raw connection strings

### Access Control
- Use separate migration credentials with limited permissions
- Restrict migration execution to authorized users
- Audit migration operations
- NO production migrations from development environments

## Best Practices

### DO
- Run migrations on startup (fail fast if they fail)
- Use transactions for schema changes when possible
- Version control all migration files
- Test migrations in staging before production
- Detect dirty state and provide recovery instructions
- Backup data before destructive operations
- Document migration purpose and breaking changes

### DON'T
- Modify existing migration files (create new ones instead)
- Run untested migrations in production
- Ignore migration errors or dirty state
- Make irreversible changes without backup plan

## Troubleshooting

### Dirty Database State
**Symptoms**: "dirty database" error, cannot apply new migrations

**Recovery**:
1. Check logs for failure details
2. Verify database state manually
3. If partially succeeded: `migrate force <version>`
4. If failed completely: `migrate down 1`
5. Fix issue and retry

### Connection Failures
**Symptoms**: "connection refused", "no such host", timeouts

**Solutions**:
- Verify database is running
- Check connection parameters
- Verify network connectivity
- Increase connection timeout

### Migration Lock Timeout
**Symptoms**: "lock timeout", migration hangs

**Solutions**:
- Check for long-running migrations
- Increase lock timeout in config
- Verify no other migration processes running

## Quick Reference

### Key Implementation Points
1. **MigrationManager** (`manager.go`) - Interface with Up/Down/Version/Close methods
2. **MigrationConfig** (`config.go`) - Validation and database URL construction
3. **RunMigrations** (`runner.go`) - Startup function with retry and dirty state detection
4. **MigrationSource** (`source.go`) - Filesystem and embedded source implementations
5. **Logger Adapter** (`logger.go`) - Bridge to golang-migrate logger interface

### Critical Rules
- Run migrations on startup before database connection
- Fail fast on errors with recovery instructions
- Sanitize passwords in ALL logs and errors
- Detect dirty state before execution
- Retry connection errors (max 3 attempts)
- NO business logic in migrations
- NO domain dependencies
